# 智企通 MVP 实现方案（v5 · 7天冲刺版）

## Context

BOCHK 2026 初赛计划书截止 6/15。
**7 天开发 MVP + 2 天 SME 测试**，成果嵌入计划书。

**MVP 范围**：Step 1（企业画像）+ Step 4（服务匹配）+ Step 5（ESG 缺口识别）
**交付**：截图 + 操作视频 + SME 试用数据 → 嵌入计划书

**策略**：先按完整版冲，时间不够再砍。预设好降级清单，到时快速裁剪。

### v5 更新要点（基于 6/6 讨论）

1. **ESG 数据架构升级**：JSON 索引 + MD 详细内容（混合格式），取代纯 JSON
2. **双标准对照**：目的地法规（企业端）+ BOCHK taxonomy（银行端），两套评分体系
3. **新增目标地区**：欧盟 CBAM 碳关税（香港/新加坡/泰国/欧盟 4 选 1）
4. **去掉 FAISS RAG**：数据量 < 100 条，直接全传 prompt，省掉复杂依赖
5. **可信度框架**：原文/AI 分析/建议三层分离 + 置信度标注
6. **引用追踪机制**：MD 文件中写文件路径 → 后端自动加载关联内容
7. **LLM 分工**：预处理用 Claude（开发环境），用户端分析用 GLM-4-Flash（运行时）

## 五、ESG 数据架构（核心变更）

### 设计理念

借鉴 Karpathy LLM Wiki 模式的核心思路：**不让 LLM 在回答时理解原始文本，而是预先消化成结构化知识。**

```
原始法规文本（唐一杰收集）
       ↓ Claude 预处理（开发环境）
结构化 MD 文件（检查项 + 评估标准 + 常见差距 + 引用关联）
       ↓ 后端加载
JSON 索引筛选 + MD 内容全传 prompt
       ↓ GLM/Qwen 分析
可信度分层的缺口报告
```

### JSON 索引文件格式

```json
[
  {
    "id": "eu_cbam_001",
    "detail_file": "details/eu_cbam_001.md",
    "region": "欧盟",
    "regulation": "CBAM",
    "category": "E",
    "title": "碳排放报告义务",
    "applicable_industries": ["钢铁", "铝", "水泥", "化肥"],
    "effective_date": "2026-01-01"
  }
]
```

### MD 详细文件格式

```markdown
---
id: eu_cbam_001
region: 欧盟
regulation: CBAM
category: E
title: 碳排放报告义务
effective_date: 2026-01-01
source: EU Regulation 2023/956 Art.8
source_url: https://...
---

## 合规检查项
- [ ] 是否有碳排放数据收集体系
- [ ] 是否能区分直接排放(Scope 1)和间接排放(Scope 2)
- [ ] 是否每季度按时向CBAM登记系统申报

## 评估标准
- 满足：已有碳审计报告 + 季度申报流程
- 部分满足：有能耗数据但未区分排放类型或未按季度申报
- 不满足：无任何碳排放相关数据收集

## SME 常见差距
多数中小制造企业缺乏碳排放监测体系，建议优先从用电量数据开始，
逐步建立 Scope 1（直接排放）和 Scope 2（间接排放）的分开记录。

## 法规原文摘要
根据 EU 2023/956 第8条，进口商应在每季度结束后一个月内，
通过 CBAM 过渡期登记系统提交其进口货物的 embedded emissions 数据...

## 相关法规
- 碳排放核查要求详见 details/eu_cbam_003.md
- 类似的季度报告要求见 details/hkma_esg_002.md
```

### 引用追踪机制

后端读取 MD 文件时，自动追踪引用：

```python
def load_with_references(md_path):
    """读取主文件 + 自动追踪引用"""
    content = read_md(md_path)

    # 从"相关法规"章节提取文件引用
    refs = extract_file_references(content)  # 匹配 details/xxx.md

    # 加载引用文件（只取摘要部分，不递归）
    ref_contexts = []
    for ref_path in refs:
        ref_content = read_md(ref_path)
        ref_contexts.append(f"[引用：{ref_path}]\n{ref_content[:500]}")  # 截断防过长

    return content, ref_contexts
```

## 六、双标准对照设计

### 目的地法规视角（企业端）

回答："我去这个国家合规吗？"

```
企业 ESG 现状 → 对照目标国家法规 → 生成缺口报告
```

数据源：`esg/index/{country}.json` + `esg/details/{country}_xxx.md`

### BOCHK 准入视角（银行端）

回答："我能拿到绿色贷款吗？"

```
企业 ESG 现状 → 对照 BOCHK 绿色分类标准 → 生成准入评估
```

数据源：`esg/index/bochk_taxonomy.json` + `esg/details/bochk_gf_xxx.md`

### 前端展示

ESG 报告页面顶部加 tab 切换：

```
[🌍 目的地法规合规]  [🏦 中银绿色贷款准入]
```

两套独立的评分和缺口列表，共享同一份问卷答案，只是对照的标准不同。

## 七、可信度框架

### UI 分层展示

每个 ESG 缺口条目分为三个视觉层：

```
┌─────────────────────────────────────┐
│ 📋 法规原文                          │  白色背景，左侧竖线引用
│ "根据EU 2023/956第8条，进口商应..."    │  带来源标注
│ 来源：EU CBAM Regulation, Art.8      │
├─────────────────────────────────────┤
│ 🤖 AI 判断：不满足                    │  浅蓝背景
│ 置信度：高（法规原文有明确要求）        │  🔵 高 / 🟡 中 / 🟠 低
├─────────────────────────────────────┤
│ 💡 AI 建议                           │  浅绿背景
│ 建议优先建立Scope 1排放数据收集...     │
│ 置信度：中（基于行业通用实践）          │
└─────────────────────────────────────┘
```

### 置信度定义

| 级别 | 含义 | 触发条件 |
|------|------|---------|
| 🔵 高 | 法规原文有明确要求，判断基于事实 | MD 文件中评估标准明确匹配 |
| 🟡 中 | 基于行业通用实践，非法规直接要求 | LLM 基于常识推断 |
| 🟠 低 | AI 推断，建议咨询专业顾问 | 信息不足或法规表述模糊 |

### 免责声明

每份 ESG 报告底部：

> ⚠️ 以上分析由 AI 生成，法规原文引用供参考，AI 判断和建议仅供参考。
> 建议咨询专业 ESG 顾问获取正式合规评估。
